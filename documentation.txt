================================================================================
PLUS33 ERP — SYSTEM DOCUMENTATION
================================================================================

This document contains the official technical and functional documentation for
the PLUS33 ERP platform. It is maintained for developers, DevOps engineers,
QA engineers, business analysts, and future system maintainers.

--------------------------------------------------------------------------------
1. PROJECT OVERVIEW
--------------------------------------------------------------------------------
* Platform Name: PLUS33 ERP
* Description: A robust, multi-tenant capable Enterprise Resource Planning (ERP)
  system designed to model organizational structures, manage workforce
  schedules and payroll, track real-time inventory levels, control procurement 
  and supply chain workflows, and perform finance accounting ledgers.
* High-Level Architecture: Layered Monolith Architecture following standard
  Spring Boot conventions: Controller -> DTO -> Mapper -> Service ->
  Repository -> JPA Entity.
* Technology Stack:
  - Java Version: JDK 17
  - Backend Framework: Spring Boot 4.0.8-SNAPSHOT, Spring Security, Spring Data JPA
  - Database: PostgreSQL (for production, staging, and integration test execution)
  - Database Migrations: Flyway (with postgresql-specific extensions)
  - Mapping: MapStruct 1.6.3
  - Lombok for boilerplate minimization
  - Nimbus JOSE JWT for stateless JWT decoding and signature verification
  - Testing: JUnit 5, MockMvc, AssertJ, Hamcrest Matchers
  - Build System: Apache Maven (wrapper scripts provided)

--------------------------------------------------------------------------------
2. MODULE OVERVIEW
--------------------------------------------------------------------------------
The application is structured into the following operational domains:

* Authentication & Authorization:
  Secures APIs using stateless JWT authentication. Resolves user roles and maps
  granted authority strings to controllers for method-level security checking.

* Organization Management:
  Represents company hierarchy and nodes:
  - Companies: Top-level logical tenant boundaries.
  - Regions: Geographical group boundaries within a company.
  - Warehouses: Physical storage nodes containing bulk inventory.
  - Stores: Retail outlets that pull stock from default warehouses.

* Workforce Management:
  Manages employee profiles, work shifts, daily clock-in/out attendance logs,
  leave type policies, leave applications, payroll periods, and calculations
  supporting overtime multipliers.

* Inventory Management & Traceability:
  Supports product catalog metadata, real-time stock balances (physical quantity and reserved quantity),
  audit trails of stock movements, manual stock corrections, and physical stock count cycle sessions.
  Also includes traceability: batch/lot expiry tracking, serial number tracking, genealogy movement logs,
  and dedicated quarantine/recall workflows.

* Procurement:
  Streamlines supply chain interactions from Purchase Requests (PRs), Supplier
  information directories, Purchase Orders (POs), and Goods Receipts (GRs)
  validating actual received physical quantities against orders.

* Finance & Ledger:
  Implements double-entry ledger bookkeeping. Defines the Chart of Accounts (COA)
  mapping codes (e.g., Accounts Payable 2100, Cash 1100, Bank 1200), Journal Entries
  representing balanced debits/credits, Supplier Invoices with three-way matching,
  and Supplier Payments allocating cash offsets.

* Analytics & Reporting:
  Uses PostgreSQL Materialized Views to perform concurrent, high-performance updates
  on key metrics like PO fulfillment, aging payables, supplier performance, and
  three-way matching exceptions. Also provides inventory dashboards for KPIs,
  stock aging/expiry, ABC/XYZ classifications, slow/dead stock, replenishment rule coverage,
  and recalls.

* Sales & Customer Management:
  Supports customer accounts (B2B and B2C), billing/shipping addresses,
  tax profile categories, credit limit thresholds, outstanding balances,
  pricing tiers, payment terms, and status lifecycles.

5. API DOCUMENTATION
--------------------------------------------------------------------------------
This section lists primary REST endpoints, parameters, permissions, validation rules,
workflow transitions, and possible errors:

* Authentication (AuthController)
  - Endpoint: POST /api/v1/auth/login
  - Purpose: Validates email/password credentials and issues a JWT token.
  - Required Permission: None
  - Request: LoginRequest DTO (email, password)
  - Response: TokenResponse (token, email, roles)
  - Errors: 400 Bad Request (validation errors), 401 Unauthorized (invalid credentials)

* Supplier Invoices (SupplierInvoiceController)
  - Endpoint: POST /api/v1/supplier-invoices
  - Purpose: Creates a draft supplier invoice linked to a PO.
  - Required Permission: SUPPLIER_INVOICE_CREATE
  - Request: SupplierInvoiceRequest DTO
  - Response: ApiResponse<SupplierInvoiceResponse>
  - Errors: 400 (validation), 404 (not found), 409 (conflict)

  - Endpoint: PUT /api/v1/supplier-invoices/{id}
  - Purpose: Updates a draft supplier invoice.
  - Required Permission: SUPPLIER_INVOICE_UPDATE
  - Request: SupplierInvoiceUpdateRequest DTO
  - Response: ApiResponse<SupplierInvoiceResponse>
  - Errors: 400 (invalid state/validation), 404 (not found)

  - Endpoint: POST /api/v1/supplier-invoices/{id}/approve
  - Purpose: Approves a draft invoice. Transitions status to APPROVED and automatically generates posted ledger journal entries.
  - Required Permission: SUPPLIER_INVOICE_APPROVE
  - Request: None
  - Response: ApiResponse<SupplierInvoiceResponse>
  - Errors: 400 (not DRAFT), 404 (not found)

  - Endpoint: POST /api/v1/supplier-invoices/{id}/cancel
  - Purpose: Cancels a supplier invoice and automatically generates reversing journal entries.
  - Required Permission: SUPPLIER_INVOICE_CANCEL
  - Request: None
  - Response: ApiResponse<SupplierInvoiceResponse>
  - Errors: 400 (not APPROVED), 404 (not found)

* Payment Processing (PaymentController)
  - Endpoint: POST /api/v1/payments
  - Purpose: Creates a completed payment, records allocations against outstanding supplier invoices, and generates posted cash/AP journal entries.
  - Required Permission: PAYMENT_CREATE
  - Request: PaymentRequest DTO (companyId, supplierId, allocations, paymentMethod, amount, referenceNumber)
  - Response: ApiResponse<PaymentResponse>
  - Errors: 400 (validation, allocation limits exceeded), 404 (not found), 409 (duplicate reference)

  - Endpoint: POST /api/v1/payments/{id}/cancel
  - Purpose: Cancels a payment, restores invoice outstanding balances, and generates reversing journal entries.
  - Required Permission: PAYMENT_CANCEL
  - Request: PaymentCancelRequest DTO (reason)
  - Response: ApiResponse<PaymentResponse>
  - Errors: 400 (not COMPLETED), 404 (not found)

* Inventory Transfers (InventoryTransferController)
  - Endpoint: POST /api/v1/inventory-transfers
  - Purpose: Initiates a draft internal stock transfer request.
  - Required Permission: INVENTORY_TRANSFER_CREATE
  - Request: InventoryTransferRequest DTO (companyId, source, destination, items)
  - Response: ApiResponse<InventoryTransferResponse>
  - Errors: 400 (validation, store-to-warehouse transfer, identical locations)

  - Endpoint: POST /api/v1/inventory-transfers/{id}/submit

6. WORKFLOW DOCUMENTATION
--------------------------------------------------------------------------------
This section defines business states, transitions, validations, and side effects.

* Purchase Requisition Workflow:
  - DRAFT: Initial creation. Editable.
  - SUBMITTED: Locks edits. Submitted for manager approval.
  - APPROVED: Manager validates items and signs off. Snapshots approved quantities.
  - CONVERTED_TO_PO: Purchase Request converted into a PO draft, linking entities.
  - REJECTED/CANCELLED: Terminal rejection states. Rejection reasons are audited.

* Purchase Order Workflow:
  - DRAFT: Initial PO.
  - ISSUED: PO approved. Expected delivery date must be present and future.
  - PARTIALLY_RECEIVED: Goods receipts contain quantities less than ordered.
  - RECEIVED: Goods receipt fully matched. Remaining quantity is 0.
  - CLOSED/CANCELLED: Terminal states.

* Goods Receipt Process:
  - Validations:
    - Linked PO must be ISSUED or PARTIALLY_RECEIVED.
    - Product must exist in the PO lines.
    - Received quantity cannot exceed remaining quantity on the PO line.
  - Side Effects:
    - Completing increments product physical on-hand quantity in inventory_stock.
    - Generates a GOODS_RECEIPT type StockMovement entry.
    - Decrements PO line remaining_quantity, increments received_quantity.
    - Cancelling a Goods Receipt reverses the entire stock addition and PO counts.

* Inventory Transfer Workflow:
  - DRAFT -> SUBMITTED -> APPROVED -> IN_TRANSIT -> RECEIVED -> CLOSED
  - Validation: Same-company constraint, store-to-warehouse transfer block.
  - Side Effects:
    - Approve: Verifies available stock (on_hand - reserved >= requested). Updates
      reserved_quantity on source stock. Publishes TRANSFER_RESERVE StockMovement.
    - Dispatch: Decrements physical quantity and reserved_quantity from source.
      Publishes TRANSFER_OUT StockMovement.
    - Receive: Increments physical quantity at destination. Marks received_quantity.
      Publishes TRANSFER_IN StockMovement. Transitions immediately to CLOSED.
    - Cancel: Reverses reserved_quantity on source. Publishes TRANSFER_RELEASE.

* Inventory Adjustment Workflow:
  - DRAFT -> SUBMITTED (if reductions exist) -> APPROVED -> POSTED
  - DRAFT -> APPROVED (auto-approved if additions only) -> POSTED
  - Validation: XOR location, mismatched company, negative stock check.
  - Side Effects:
    - Post: Locks stock and adjusts physical quantity by the delta. Publishes INVENTORY_ADJUSTMENT StockMovement. Fires InventoryAdjustmentRefreshEvent.

* Stock Count & Cycle Count Workflow:
  - DRAFT -> ASSIGNED -> IN_PROGRESS -> SUBMITTED -> APPROVED -> POSTED -> CLOSED
  - Exception Path: SUBMITTED -> REJECTED -> IN_PROGRESS (via reopen)
  - Validation: XOR location, mismatched company, overlapping active count check.
  - Blind Count Rule: Hide system quantity and variance details from DTO responses while active (DRAFT, ASSIGNED, IN_PROGRESS).
  - Side Effects:
    - Start: Snapshots current stock levels and takes immutable quantities.
    - Submit: Computes variances. Auto-approves directly to APPROVED if all lines are within threshold.
    - Reopen: Resets counted quantity and variance. Increments recount_number.
    - Approve: Automatically creates STOCK_COUNT_VARIANCE InventoryAdjustment.
    - Post: Triggers posting on the variance InventoryAdjustment.

* Inventory Replenishment Workflow:
  - Rules: DRAFT -> ACTIVE
  - Suggestions: OPEN -> ACKNOWLEDGED -> ORDERED -> FULFILLED
  - Suggestions alternate: OPEN/ACKNOWLEDGED -> CANCELLED
  - Validation:
    - Rule location must target exactly one warehouse or store.
    - Rule quantities constraints: min >= 0, point >= min, max > point, reorder > 0, reorder <= max.
    - Locations and parent rules must belong to same company.
    - Only one active (OPEN/ACKNOWLEDGED) suggestion per rule.
  - Side Effects:
    - Evaluation: Compares available quantity against reorder point. Creates OPEN suggestion if at or below reorder point.
    - Create PR: Auto-creates a draft Purchase Request using first active company supplier and links it. Transitions to ORDERED.
    - Create Transfer: Only for stores. Auto-creates a draft Inventory Transfer using first active company warehouse as source and links it. Transitions to ORDERED.

* Inventory Traceability & Recall Workflows:
  - FEFO Allocation: Lots are queried sorted by expiry_date ASC, optimizing allocation to limit waste.
  - Serial Status Lifecycle: State transitions are strictly validated:
    - IN_STOCK -> TRANSIT / SOLD / RECALLED / QUARANTINED (Valid)
    - TRANSIT / QUARANTINED -> IN_STOCK (Valid)
    - SOLD / RECALLED are terminal statuses and cannot transition further.
  - Automated Expiry Scheduler: A nightly background job runs at 01:15 AM, checks expired lots (expiry_date < today), updates status to EXPIRED, and records auto-expiry events.
  - Recall Quarantine: Creating a recall isolates the target lot/serial, updates statuses to RECALLED, and generates audit trail events. Unique indices block overlapping recalls.

* Customer Management Workflow:
  - Customer status: ACTIVE, INACTIVE, ON_HOLD, BLOCKED.
  - CRUD operations are restricted to company scope (no cross-company manipulation).
  - Validation: checks on credit_limit >= 0, outstanding_balance >= 0, discount_rate between 0 and 100, and payment_terms_days >= 0.
  - Customer Code Generation: Generated automatically on creation using format CUST-YYYY-xxxxxx.
  - Soft Delete: soft deletes are performed by updating status to INACTIVE.

9. ANALYTICS DOCUMENTATION
--------------------------------------------------------------------------------
* Materialized Views:
  - mv_procurement_summary: Company metrics.
  - mv_supplier_performance: On-time delivery rates and lead times.
  - mv_payables_aging: Groups unpaid invoices by aging buckets (current, 30, 60, 90+ days).
  - mv_po_fulfilment: Order lines fulfillment status.
  - mv_invoice_matching: Flags three-way matching discrepancies (unit price/quantity).
  - mv_inventory_kpis: Aggregates total stock value, unique product counts, out of stock, low stock, and overstocks.
  - mv_inventory_aging_expiry: Calculates stock aging buckets (0-30, 31-90, 91-180, 180+ days) and lot expiry risks.
  - mv_inventory_abc_xyz: Performs ABC/XYZ demand volatility classifications over 365 days.
  - mv_inventory_slow_dead: Identifies slow-moving and dead stock items.
  - mv_inventory_replenishment_metrics: Tracks rules coverage, cycle times, and stockout prevention rates.
  - mv_inventory_traceability_metrics: Audits recalls, serial counts, and product compromise rates.
  - mv_inventory_turnover: Calculates COGS, average stock value, and turnover ratios.
* Indexes for Concurrent Refreshes:
  Unique indexes (e.g. uidx_mv_payables_aging, uidx_mv_inventory_kpis) allow non-blocking concurrent refreshes
  ("REFRESH MATERIALIZED VIEW CONCURRENTLY view_name"). Expression indexes cannot be used for concurrent refreshes;
  thus, unique indices are created over plain columns (company_id, warehouse_id, store_id) directly.
* Scheduling:
  AnalyticsRefreshService runs scheduled staggered cron updates to minimize DB contention:
  - Every 15 mins (0, 15, 30, 45): KPIs
  - Every 15 mins (5, 20, 35, 50): Aging/Expiry
  - Every 15 mins (10, 25, 40, 55): ABC/XYZ
  - Every 15 mins (12, 27, 42, 57): Traceability
  - Every 15 mins (14, 29, 44, 59): Replenishment
  - Every 15 mins (7, 22, 37, 52): Turnover

12. TROUBLESHOOTING GUIDE
--------------------------------------------------------------------------------
* Optimistic Locking Failures (ObjectOptimisticLockingFailureException):
  Occur when two threads edit the same entity (version mismatch). Resolve by implementing
  optimistic retry wrappers or locking via PESSIMISTIC_WRITE.
* Unique Constraint Violations:
  Often due to missing @Transactional in integration tests or missing flush() after deletes in @BeforeEach.
  Resolve by verifying transactional rollback and setup flush.
* Hibernate Flush Ordering:
  When using clear() + addAll() on collection replacements, Hibernate executes inserts before deletes.
  Declaring composite uniqueness constraints as DEFERRABLE INITIALLY DEFERRED is required.

--------------------------------------------------------------------------------
13. DEVELOPER GUIDELINES
--------------------------------------------------------------------------------
* Collection Update Constraint Rule:
  Whenever a collection mapping uses cascade = CascadeType.ALL and orphanRemoval = true,
  and the collection is replaced using clear() + addAll(newItems), the composite unique
  key containing the parent foreign key in the child table MUST be declared as
  "DEFERRABLE INITIALLY DEFERRED". This defers unique constraint evaluation until commit,
  preventing statement-level violations during Hibernate's insert-before-delete flush.
