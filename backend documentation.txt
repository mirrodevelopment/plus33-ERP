1. PROJECT OVERVIEW
================================================================================

PROJECT NAME:    PLUS33 ERP
ARTIFACT ID:     plus33-erp
GROUP ID:        com.plus33
VERSION:         0.0.1-SNAPSHOT
BASE PACKAGE:    com.plus33.erp

PURPOSE:
    PLUS33 ERP is a comprehensive, full-spectrum Enterprise Resource Planning
    platform built on Spring Boot. It covers every functional area of a modern
    enterprise: finance, procurement, sales, inventory, manufacturing, human
    capital management, warehouse management, field service, GRC, business
    intelligence, AI agents, digital twins, IoT/SCADA, logistics, routing,
    ESG compliance, and edge computing.

BUSINESS DOMAIN:
    Multi-company, multi-warehouse, multi-currency enterprise operations.
    The system serves large/enterprise-scale organizations requiring:
      - Full General Ledger with double-entry journal entries
      - Multi-tier procurement: PR -> RFQ -> PO -> GRN -> Invoice
      - Sales lifecycle: Orders -> Picking -> Invoicing -> Returns
      - Full WMS with RFID, robotics, wave-picking, cycle counting
      - Manufacturing: BOM, routing, MRP II, production orders, quality
      - HCM: recruitment -> payroll -> succession
      - GRC: enterprise risk, audit, SOD, policy management
      - AI cognitive engine for autonomous decision support
      - Digital twin process mining and conformance checking
      - ESG/carbon footprint tracking (Scope 1 & 2)
      - Edge computing and IoT/SCADA integration

ARCHITECTURE PHILOSOPHY:
    - Domain-Driven Design (DDD) with bounded contexts as top-level packages
    - Layered: Controller -> Service -> Repository -> Entity
    - Event-driven via Spring ApplicationEventPublisher
    - CQRS-lite: write services + read-optimized views/snapshots
    - Flyway schema evolution (376 migrations, 323 actual files)
    - JWT stateless authentication with Role + Permission RBAC
    - MapStruct DTO<>Entity transformation
    - JPA Specifications for dynamic queries

3. OVERALL ARCHITECTURE
================================================================================

LAYERED ARCHITECTURE (top to bottom):
    CLIENT (HTTP)
         |
    SECURITY FILTER CHAIN
    JwtAuthFilter -> UsernamePasswordAuthenticationFilter
         |
    REST CONTROLLERS (@RestController, @RequestMapping)
    Authorization: @PreAuthorize('hasAuthority(PERM_CODE)')
    Validation: @Valid on @RequestBody
    Response: ApiResponse<T> envelope
         |
    SERVICE LAYER (interface + @Service implementation)
    @Transactional boundaries
    Business logic, validation, event publishing
    Uses: Repositories, Mappers, EventPublisher
         |
    REPOSITORY LAYER (JpaRepository + Specification / JPQL)
    Spring Data JPA with custom @Query methods
         |
    PERSISTENCE LAYER (JPA/Hibernate -> PostgreSQL)
    @Entity classes with Flyway-managed DDL

REQUEST LIFECYCLE:
    1.  HTTP request arrives at Spring DispatcherServlet
    2.  JwtAuthFilter intercepts (OncePerRequestFilter)
        a. Skip if path starts with /api/v1/auth/, /swagger-ui/, /v3/api-docs
        b. Extract Authorization: Bearer <token>
        c. NimbusJwtDecoder decodes JWT (HMAC-SHA256)
        d. Build UsernamePasswordAuthenticationToken with authority list
        e. Set in SecurityContextHolder
    3.  Spring Security checks @PreAuthorize on controller method
    4.  Controller validates @RequestBody with @Valid -> Bean Validation
    5.  Controller calls Service method
    6.  Service executes business logic, accesses repositories
    7.  Service publishes domain events if needed
    8.  MapStruct mapper converts Entity -> DTO
    9.  Controller wraps in ApiResponse<T>
    10. GlobalExceptionHandler catches any exceptions -> ErrorResponse

AUTHENTICATION LIFECYCLE:
    POST /api/v1/auth/login
    -> AuthController.login(LoginRequest)
    -> AuthenticationManager.authenticate(email, password)
    -> DaoAuthenticationProvider -> UserDetailsServiceImpl
    -> UserRepository.findByEmail(email)
    -> Build UserDetails (email, bcrypt password, ROLE_ + PERMISSION authorities)
    -> JwtService.generateToken(UserDetails)
    -> NimbusJwtEncoder signs: issuer=plus33-erp, sub=email, authorities=list
    -> Return TokenResponse(accessToken, 'Bearer', expiresIn)

AUTHORIZATION LIFECYCLE:
    1. JWT decoded by JwtAuthFilter; 'authorities' claim extracted
    2. SimpleGrantedAuthority list set in SecurityContextHolder
    3. @PreAuthorize('hasAuthority(PERMISSION_CODE)') evaluated by SpEL
    4. AccessDeniedException -> GlobalExceptionHandler -> 403

VALIDATION LIFECYCLE:
    @Valid -> Bean Validation 3.0 -> MethodArgumentNotValidException
    -> GlobalExceptionHandler -> 400 + field error map

EXCEPTION LIFECYCLE:
    ResourceNotFoundException     -> 404
    DuplicateResourceException    -> 409
    BusinessException             -> 400
    MethodArgumentNotValidException -> 400 + field errors
    ConstraintViolationException  -> 400
    HttpMessageNotReadableException -> 400 Malformed JSON
    DataIntegrityViolationException -> 409 DB constraint
    AccessDeniedException         -> 403
    OptimisticLockException       -> 409
    Exception (fallback)          -> 500

TRANSACTION LIFECYCLE:
    - @Transactional(readOnly=true) at class level on services
    - Write methods override with @Transactional (readOnly=false)
    - AuditableEntity: @EntityListeners(AuditingEntityListener.class)
    - SecurityAuditorAware implements AuditorAware<Long>
    - @CreatedDate, @LastModifiedDate, @CreatedBy, @LastModifiedBy auto-populated

CACHING:
    Spring Cache enabled; cache infrastructure in V211 migration.
    @Cacheable/@CacheEvict on high-read service methods.

LOGGING:
    Lombok @Slf4j throughout. SLF4J -> Logback (Spring Boot default).
    JwtAuthFilter logs debug per-request path, warns on JwtException.

PACKAGE: com.plus33.erp.security

SECURITY MODEL (3-tier):
    User  -- many -->  Roles  -- many -->  Permissions
    Authorities in JWT = ROLE_<code> + all Permission codes for assigned roles
    Controllers use: @PreAuthorize('hasAuthority(PERMISSION_CODE)')

INITIAL ROLES (seeded V3):
    ULTIMATE_ADMIN   - Full system access
    REGIONAL_ADMIN   - Regional operations
    STORE_ADMIN      - Store operations
    SHIFT_SUPERVISOR - Shift management
    SENIOR_EMPLOYEE  - Senior employee
    JUNIOR_EMPLOYEE  - Junior employee
    TRAINEE          - Training employee

INITIAL PERMISSIONS (seeded V3):
    USER_VIEW, USER_CREATE, USER_EDIT
    STORE_VIEW, STORE_CREATE, STORE_EDIT
    INVENTORY_VIEW, INVENTORY_EDIT
    WORKFORCE_VIEW, WORKFORCE_EDIT
    (Additional module-specific permissions in V23-V376)

--- JwtConfig.java (@Configuration) ---
    jwtEncoder() -> NimbusJwtEncoder
        SecretKeySpec(jwt.secret.getBytes(UTF8)), HmacSHA256
        ImmutableSecret<> as JWK source
    jwtDecoder() -> NimbusJwtDecoder
        NimbusJwtDecoder.withSecretKey(secretKey).build()

--- SecurityConfig.java (@Configuration @EnableWebSecurity @EnableMethodSecurity) ---
    Order: Ordered.HIGHEST_PRECEDENCE
    securityFilterChain(HttpSecurity http):
        CSRF: disabled (stateless JWT API)
        Session: STATELESS
        permitAll paths:
            /api/v1/auth/**
            /actuator/health
            /swagger-ui/**, /swagger-ui.html
            /v3/api-docs/**, /api-docs/**
            /h2-console/**   [REMOVE IN PRODUCTION]
        anyRequest: authenticated()
        JwtAuthFilter added BEFORE UsernamePasswordAuthenticationFilter
    authenticationProvider():
        DaoAuthenticationProvider(userDetailsService, passwordEncoder)
    authenticationManager(): exposed as @Bean for AuthController

--- JwtService.java (@Service @RequiredArgsConstructor) ---
    Fields: JwtEncoder, @Value('\') long
    generateToken(UserDetails) -> signed JWT
        Claims: issuer='plus33-erp', issuedAt, expiresAt, subject=email
                authorities=List<String> of granted authorities
        Header: HS256 (MacAlgorithm.HS256)
    getExpirationSeconds() -> expirationMinutes * 60

--- JwtAuthFilter (extends OncePerRequestFilter, @Slf4j) ---
    Skip paths: /api/v1/auth/, /swagger-ui/, /v3/api-docs, /api-docs
    Extract: Authorization: Bearer <token>
    Decode: jwtDecoder.decode(token)
    Build: UsernamePasswordAuthenticationToken(subject, null, authorities)
    Set: SecurityContextHolder
    On JwtException: log.warn, continue filterChain (unauthenticated)

--- UserDetailsServiceImpl (@Service) ---
    loadUserByUsername(email):
        userRepository.findByEmail(email) or throw UsernameNotFoundException
        For each Role: add SimpleGrantedAuthority('ROLE_' + code)
        For each Permission in Role: add SimpleGrantedAuthority(code)
        Return User(email, password_bcrypt, authorities, disabled=!active)

ENTITIES:
    User (@Entity, table=users)
        id BIGSERIAL PK, email VARCHAR UNIQUE NOT NULL, password VARCHAR NOT NULL
        firstName VARCHAR(100) NOT NULL, lastName VARCHAR(100)
        active BOOLEAN DEFAULT true, createdAt, updatedAt
        roles: Set<Role> LAZY ManyToMany via user_roles

    Role (@Entity, table=roles)

33. SECURITY GUIDE
================================================================================

JWT CONFIGURATION:
    Two properties control JWT: jwt.secret (JwtConfig) and app.jwt.secret (JwtService).
    Both must be 256-bit (32+ character) random strings.
    Rotate together to avoid token decode failures.
    Never commit real secrets to version control.

TOKEN EXPIRATION:
    Default: 60 minutes. Configurable via app.jwt.expiration-minutes.
    No refresh token mechanism. Users re-authenticate when token expires.
    Consider implementing refresh tokens for production UX.

SESSION MANAGEMENT:
    SessionCreationPolicy.STATELESS - no server-side sessions.
    CSRF protection disabled (appropriate for stateless APIs).

PASSWORD STORAGE:
    BCryptPasswordEncoder. Default strength (10 rounds).
    Passwords never stored or returned in plaintext.

INITIAL ADMIN:
    Seeded in V5__seed_ultimate_admin_user.sql with ULTIMATE_ADMIN role.
    CHANGE THE DEFAULT PASSWORD immediately after first deployment.

CRITICAL PRODUCTION FIXES:
    1. spring.flyway.clean-disabled=true      (prevent schema wipe)
    2. spring.flyway.clean-on-validation-error=false
    3. Remove /h2-console/** from SecurityConfig permitAll
    4. Externalize all credentials to environment variables
    5. Change default admin password

CORS:
    Not explicitly configured. Spring Security defaults: no CORS headers.
    Add CorsConfigurationSource @Bean to SecurityConfig for browser access.

PERMISSION ENFORCEMENT:
    @PreAuthorize on every controller method.
    Missing annotation = publicly accessible = security vulnerability.
    Run security audit to verify 100% coverage.

POTENTIAL BRUTE FORCE PROTECTION:
    Not visible in analyzed code. Consider adding:
    - Account lockout after N failed attempts
    - Rate limiting on /api/v1/auth/login
    - IP-based blocking

    9.  JPA Specifications for dynamic, composable queries
    10. Flyway for reproducible, versioned schema evolution
    11. Interface + Implementation pattern throughout services
    12. Clean bounded context separation
    13. Lombok reduces boilerplate while preserving readability
    14. Spring Actuator + Prometheus for operational visibility
    15. Comprehensive integration test suite per module

ISSUES / RECOMMENDATIONS:

    [CRITICAL - SECURITY - MUST FIX BEFORE PRODUCTION]
    1. Hardcoded credentials in application.properties:
       - spring.datasource.password=crazy@8
       - jwt.secret=plus33-super-secret-key...
       - app.jwt.secret=plus33-erp-jwt-secret-key...
       Action: Externalize ALL credentials to environment variables.

    [CRITICAL - DATA SAFETY - MUST FIX BEFORE PRODUCTION]
    2. spring.flyway.clean-disabled=false
       spring.flyway.clean-on-validation-error=true
       These allow Flyway to DROP THE ENTIRE DATABASE on validation errors.
       Action: Set clean-disabled=true and clean-on-validation-error=false.

    [HIGH - SECURITY]
    3. H2 console (/h2-console/**) in permitAll list.
       Action: Remove from SecurityConfig for production builds.

    [HIGH - AUDIT GAP]
    4. SecurityAuditorAware tries to parse email (JWT subject) as Long.
       Since subjects are email strings, this always fails.
       createdBy/updatedBy are always null in AuditableEntity.
       Action: Add numeric userId claim to JWT, or use String auditor type.

    [MEDIUM - DESIGN]
    5. Dual JWT secret properties (jwt.secret vs app.jwt.secret).
       JwtConfig uses jwt.secret; JwtService uses app.jwt.secret.
       Action: Consolidate to a single property key.

    [MEDIUM - DESIGN]
    6. Very large service classes:
       CustomerReturnServiceImpl: ~43KB
       CustomerInvoiceServiceImpl: ~36KB
       StockCountServiceImpl: ~38KB
       ProductionOrderServiceImpl: ~31KB
       SupplierInvoiceServiceImpl: ~40KB (778 lines, 14 dependencies)
       Action: Extract domain sub-services for better single-responsibility.

    [MEDIUM - SECURITY]
    7. No refresh token implementation.
       Sessions end hard after 60 minutes. Users must re-authenticate.
       Action: Implement refresh token endpoint.

    [LOW - QUALITY]
    8. TestController in auth/controller/ - development endpoint.
       Action: Verify it is not accessible in production or remove.

    [LOW - DOCUMENTATION]
    9. Migration version gaps should be documented in a MIGRATIONS.md file.
       This prevents confusion for future developers.

    [LOW - OBSERVABILITY]
    10. Structured logging could be improved in services.
        Key business events (invoice approved, order shipped) should be
        logged at INFO level with structured fields for log aggregation.

44. COMPLETE BUSINESS WORKFLOW DOCUMENTATION (Section 9)
================================================================================

This section models the operational workflows step-by-step from source event to ledger entries.

--------------------------------------------------------------------------------
1. Procurement Workflow
--------------------------------------------------------------------------------
1. Purchase Request (PR) Creation:
   - User submits a PR. Status -> DRAFT.
   - User submits PR. Status -> SUBMITTED.
   - Evaluated by PolicyEngine (threshold limits check).
   - Approval routing -> PR approved. Status -> APPROVED.
2. Sourcing & RFQ:
   - RFQ created from approved PR.
   - Supplier invitation sent. Suppliers submit RFQ SupplierResponse.
   - Evaluate bids. Award RFQ.
3. Purchase Order (PO):
   - PO created from RFQ/PR. Status -> DRAFT.
   - PO submitted & approved. Status -> APPROVED.
   - Budget commitment logged in BudgetService (soft reservation).
4. Goods Receipt Note (GRN):
   - Physical delivery verified. GoodsReceipt generated.
   - Warehouse location stock incremented: on_hand_quantity.
   - PO eceived_quantity updated.
5. Supplier Invoice:
   - Supplier Invoice submitted against PO & GRN.
   - 3-way matching verified: Invoice Qty/Price matches PO and GRN limits.
   - Invoice Approved -> GL entries posted:
     - Debit Expense (or Inventory Asset)
     - Credit Accounts Payable
   - Budget commitment converted to actual spend.
6. Payment:
   - Payment processed and allocated to invoices.
   - GL Entries posted:
     - Debit Accounts Payable
     - Credit Cash/Bank

--------------------------------------------------------------------------------
2. Sales Workflow
--------------------------------------------------------------------------------
1. Customer Account & Credit:
   - B2B customer registered with creditLimit.
2. Sales Order (SO):
   - User enters SO. Credit limit evaluated.
   - SO confirmed. Status -> CONFIRMED.
3. Inventory Allocation:
   - Soft reservation registered in WMS: eserved_quantity is incremented in inventory_stock matching the SO items.
4. Picking & Wave management:
   - Waves grouped. PickList created.
   - Picker completes PickList. WMS Stock movement logs recorded.
5. Shipping:
   - Delivery items packaged and confirmed -> Shipment generated.
   - WMS deducts eserved_quantity and on_hand_quantity.
6. Customer Invoice:
   - Invoice generated. Status -> ISSUED.
   - GL Entries posted:
     - Debit Accounts Receivable
     - Credit Revenue
7. Payment Receipts:
   - Cash receipt allocated -> Debit Cash/Bank, Credit Accounts Receivable.

--------------------------------------------------------------------------------
3. Manufacturing Workflow
--------------------------------------------------------------------------------
1. MRP II Planning:
   - Scheduled MRP run detects net shortages from Sales Orders/Forecasts.
   - MRP explodes BOMs and schedules Planned Orders.
2. Production Order:
   - Planned order firmed to ProductionOrder. Status -> RELEASED.
3. Material Issue:
   - Raw materials transferred to work center (ProductionMaterialIssue).
   - Stock deducted from warehouse locations, added to WIP.
4. Operations & Labor:
   - Steps verified on RoutingOperations. Labor hours recorded.
5. Production Confirmation (Receipt):
   - Finished goods received from work center (ProductionConfirmation).
   - Standard costing applied -> variance computed in ProductionCost.
   - FG stock incremented in target warehouse location. WIP deducted.

--------------------------------------------------------------------------------
4. Workforce & Payroll Workflow
--------------------------------------------------------------------------------
1. Recruitment:
   - JobRequisition created -> Candidate applied -> hired -> Employee created.
2. Shift Planning & Scheduling:
   - ShiftPattern configured -> Roster generated.
3. Attendance:
   - Clock-in/out registers attendance logs.
4. Payroll Processing:
   - Monthly payroll runs evaluate attendance, leave deductions, and overtime parameters.
   - Exporter generates SEPA/ACH banking files.
   - Payslips published via EmployeeSelfService.

47. COMPLETE SECURITY DOCUMENTATION (Section 12)
================================================================================

This section outlines stateless authentication, filter execution chains, security context details, token lifecycles, and passwords handling.

* Cryptographic Algorithm: HMAC-SHA256 (HS256).
* Token Issuance & Filters Execution Order:
  1. Client sends POST to /api/v1/auth/login.
  2. Credentials verified by DaoAuthenticationProvider using BCrypt.
  3. JwtService generates signed token.
  4. Subsequent requests pass Authorization: Bearer <token>.
  5. JwtAuthFilter intercepts -> decodes JWT using NimbusJwtDecoder -> checks signature -> extracts "authorities" claim.
  6. SimpleGrantedAuthorities list is loaded.
  7. UsernamePasswordAuthenticationToken registered in SecurityContextHolder.getContext().
  8. @PreAuthorize checks SpEL expressions against authorities.
  9. If invalid, exception is thrown -> caught by GlobalExceptionHandler -> 403 response.

* Password Encryption:
  - Uses BCryptPasswordEncoder configured as a Spring bean in SecurityBeansConfig.
  - Default strength parameter: 10 rounds of salt generation.
  - Plaintext credentials are never stored.

  - Origins: Save duplicate unique indexes.
  - Handler: GlobalExceptionHandler.
  - HTTP Status: 409 Conflict.
  - Recovery: Generate unique code prefix and retry.

* OptimisticLockException:
  - Origins: Simultaneous transaction updates to the same PO or order version.
  - Handler: GlobalExceptionHandler.
  - HTTP Status: 409 Conflict.
  - Recovery: Reload latest entity status from DB and merge changes.

60. COMPLETE DEVELOPER GUIDE (Section 25)
================================================================================

This guide defines coding standards, naming conventions, and workflows for extending the PLUS33 ERP platform.

* Coding Standards:
  - Layer separation: Controllers must not access repositories directly (always go through Services).
  - Business logic must reside in Service implementation classes (ServiceImpl).
  - Return standardized envelopes (ApiResponse<T>) from REST controllers.
  - Enable optimistic locking (@Version) on transactional entities.
  - Disable Open Session in View (OSIV) - fetch required associations using FETCH JOIN in repositories.

* Naming Conventions:
  - Entities: Singular CamelCase (e.g. SupplierInvoice).
  - Repositories: EntityName + Repository (e.g. SupplierInvoiceRepository).
  - Services: EntityName + Service / EntityName + ServiceImpl.
  - DTOs: Request/Response suffix (e.g. SupplierInvoiceRequest).
  - Tables: Plural lower_snake_case (e.g. supplier_invoices).

* How to Add a Bounded Context Module:
  1. Create a top-level package in com.plus33.erp: com.plus33.erp.newmodule.
  2. Sub-packages: controller, service, service.impl, epository, entity, dto, mapper.
  3. Create SQL migrations in src/main/resources/db/migration using next Flyway version prefix (e.g., V377__create_new_module.sql).
  4. Write JPA Entity and Repository interfaces.
  5. Code Service interface and Implementation. Annotate write methods with @Transactional.
  6. Code REST Controller and annotate with @RestController, @RequestMapping("/api/v1/new-module").
  7. Add authorization check annotations: @PreAuthorize("hasAuthority('NEW_MODULE_EDIT')").
  8. Write integration test suite: NewModuleIntegrationTest.java in src/test/java.
  9. Verify compile & check validations: .\mvnw.cmd clean verify.

Low-level code processing flows, transaction boundaries, and DB modifications for service methods:

--------------------------------------------------------------------------------
1. SupplierInvoiceServiceImpl.approveInvoice(Long id)
--------------------------------------------------------------------------------
* Method Signature: public SupplierInvoiceResponse approveInvoice(Long id)
* Purpose: Approves a draft supplier invoice, validates matching quantities, and posts journal entries.
* Parameters: id (Long, invoice identifier)
* Return Type: SupplierInvoiceResponse
* Transaction: @Transactional(readOnly = false) (Atomic transaction rollback on runtime exception).
* Validation:
  - Verifies invoice existence (throws ResourceNotFoundException if absent).
  - Assures current invoice status equals DRAFT or SUBMITTED.
  - Executes 3-way matching validation: line item quantities must not exceed matching PO item quantities minus already invoiced quantities.
* Repository Calls:
  - SupplierInvoiceRepository.findById(id)
  - PurchaseOrderItemRepository.save() (increments invoicedQuantity per line)
  - JournalEntryRepository.save() (registers debit/credit postings)
* Mapper Calls: SupplierInvoiceMapper.toResponse(SupplierInvoice)
* Events Published: SupplierInvoiceRefreshEvent and ProcurementRefreshEvent.
* Exceptions: ResourceNotFoundException (404), BusinessException (400) if status transitions are illegal.
* Processing Flow:
  1. Fetch invoice entity. Check status.
  2. Validate quantities against PurchaseOrder lines.
  3. Invoke TaxCalculationEngine to compute VAT/GST adjustments.
  4. Create JournalEntry header (status = POSTED).
  5. Create JournalEntryLine 1: Debit Expense Account (5000) or WIP (1400) for net amount.
  6. Create JournalEntryLine 2: Debit Tax Receivable Account (1350) for tax amount.
  7. Create JournalEntryLine 3: Credit Accounts Payable Account (2100) for total amount.
  8. Save JournalEntry and update PO invoiced quantities.
  9. Set invoice status to APPROVED. Save invoice.
  10. Publish application events. Return mapped DTO response.

CHAPTER 30. FRONTEND INTEGRATION GUIDE
================================================================================

This guide defines the integration path for client applications interfacing with the PLUS33 ERP API.

--------------------------------------------------------------------------------
1. Authentication & JWT Storage
--------------------------------------------------------------------------------
* Token Retrieval: Client posts username and credentials to /api/v1/auth/login.
* Storage Best Practice: Store JWT token in secure client-side cookie (HttpOnly, Secure, SameSite=Strict) or in memory variable context. Avoid storing sensitive tokens in LocalStorage to reduce XSS vulnerabilities.
* Authorization Header: Set header on every REST API call:
  Authorization: Bearer <accessToken>

--------------------------------------------------------------------------------
2. Standard Response Patterns
--------------------------------------------------------------------------------
* Success Response (ApiResponse):
  {
    "success": true,
    "message": "Resource processed successfully",
    "data": { ... },
    "timestamp": "2026-07-02T10:00:00"
  }
* Error Response (ErrorResponse):
  {
    "status": 400,
    "error": "Bad Request",
    "message": "Field errors detected",
    "path": "/api/v1/supplier-invoices",
    "fieldErrors": {
      "invoiceNumber": "must not be blank"
    },
    "timestamp": "2026-07-02T10:01:00"
  }

--------------------------------------------------------------------------------
3. Query Conventions (Pagination, Filter, Sort)
--------------------------------------------------------------------------------
* List endpoints utilize page parameters:
  - page: 0-indexed page number.
  - size: page limit size (default 10).
  - sort: field name key.
  - direction: asc or desc.
  GET /api/v1/supplier-invoices?page=0&size=20&sort=invoiceDate&direction=desc
* Date Formats: Client must send dates formatted as ISO-8601 strings:
  YYYY-MM-DD (for LocalDate parameters)
  YYYY-MM-DDTHH:mm:ss.SSSZ (for LocalDateTime parameters)
